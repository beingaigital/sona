## 4. 核心文件功能说明

### 4.1 `definitions.py`

位置：

```text
opinion_analysis_kb/schema/definitions.py
```

功能：

- 定义知识库 Schema v2.0 的核心 Pydantic 模型。
- 提供所有知识类型共享的统一元数据字段。
- 提供知识类型映射 `SCHEMA_TYPES`。
- 提供 `validate_and_parse()` 用于结构化 JSON 校验。

当前定义了 8 种知识类型：

```text
Methodology
Case
DomainPlaybook
Actor
RiskPattern
ResponseTactic
Evidence
Scenario
```

其中原任务要求的 6 种类型已全部覆盖：

```text
Methodology
Case
DomainPlaybook
Actor
RiskPattern
ResponseTactic
```

额外新增：

| 新增类型 | 作用 |
|---|---|
| Evidence | 证据片段，支持报告可追溯 |
| Scenario | 舆情场景，支持更细粒度的领域划分和案例匹配 |

---

### 4.2 统一元数据字段设计

所有知识类型均继承 `BaseKnowledge`，统一包含：

```text
id
type
domain
tags
source_path
source_url
evidence_refs
confidence
status
created_at
updated_at
last_verified_at
```

设计目的：

- `id`：支持 JSON、Neo4j 节点和关系引用。
- `type`：标识知识对象类型。
- `domain`：区分健康、交通、大熊猫等垂类。
- `source_path` / `source_url`：保留原始材料来源。
- `evidence_refs`：支持证据追溯。
- `confidence`：支持置信度控制和 Graph RAG 排序。
- `status`：支持 raw、candidate、approved、deprecated 的知识状态管理。
- `created_at` / `updated_at` / `last_verified_at`：支持后续版本管理和人工审核。

`confidence` 已升级为数值型：

```text
0.0 - 1.0
```

并兼容旧格式：

```text
high -> 0.9
medium -> 0.6
low -> 0.3
```

### 4.3 `compiler.py`

位置：

```text
opinion_analysis_kb/schema/compiler.py
```

功能：

- 读取 markdown 文件。
- 解析 YAML frontmatter。
- 提取二级标题章节。
- 根据标题、frontmatter 和内容进行知识类型分类。
- 将 markdown 转为结构化 JSON。
- 使用 Pydantic Schema 校验结构化结果。
- 批量编译 wiki 目录。
- 支持 `--clean` 清理旧编译结果。
- 生成编译报告、错误报告和字段覆盖率报告。

核心函数：

| 函数 | 作用 |
|---|---|
| `parse_markdown()` | 解析 markdown frontmatter 和正文 |
| `extract_sections()` | 提取 markdown 二级标题章节 |
| `parse_list_field()` | 从章节中解析列表字段 |
| `classify_by_content()` | 判断知识类型 |
| `markdown_to_structured()` | 单个 markdown 转结构化 JSON |
| `compile_wiki_directory()` | 批量编译目录 |
| `generate_field_coverage_report()` | 生成增强字段覆盖率报告 |

## 5. 强化核心类型设计

### 5.1 强化 Case

新增字段：

```text
trigger_points
key_claims
public_sentiment
escalation_path
turning_points
official_response
resolution
lessons_learned
```

设计目的：

| 字段 | 解决的问题 |
|---|---|
| `trigger_points` | 事件为什么爆发 |
| `key_claims` | 网民主要质疑什么 |
| `public_sentiment` | 情绪结构是什么 |
| `escalation_path` | 舆情如何升级 |
| `turning_points` | 哪些节点改变了舆情走向 |
| `official_response` | 官方或责任方如何回应 |
| `resolution` | 事件最终如何处理 |
| `lessons_learned` | 有什么经验可复用 |

当前旧 wiki 编译结果中，部分 Case 增强字段已有非空内容。旧文档由于不是按 Schema v2.0 编写，因此填充率有限，后续领域资料会通过模板显式填写。

---

### 5.2 强化 RiskPattern

新增字段：

```text
trigger_conditions
early_warning_signals
typical_escalation_path
recommended_tactics
```

设计目的：

| 字段 | 解决的问题 |
|---|---|
| `trigger_conditions` | 什么情况算风险 |
| `early_warning_signals` | 早期信号是什么 |
| `typical_escalation_path` | 风险会如何升级 |
| `recommended_tactics` | 应该用什么策略应对 |

该设计让风险模式从“描述型知识”升级为“可预警、可匹配、可行动”的结构化知识。

---

### 5.3 强化 ResponseTactic

新增字段：

```text
stage
actions
message_templates
do_and_dont
success_metrics
failure_modes
```

设计目的：

| 字段 | 解决的问题 |
|---|---|
| `stage` | 分阶段行动，例如 0-6h、6-24h |
| `actions` | 具体要做什么 |
| `message_templates` | 可以怎么说 |
| `do_and_dont` | 能做什么、不能做什么 |
| `success_metrics` | 如何判断策略有效 |
| `failure_modes` | 哪些情况下可能失败 |

该设计让回应建议从“泛泛建议”升级为“可执行行动清单”。

## 6. 旧 wiki 弱映射设计

现有 wiki 文档主要遵循旧版八段式结构：

```text
事件概述
关键事实
传播机制
情绪与立场
风险点
可复用方法论
相似议题线索
引用片段
```

为了兼容旧文档，编译器实现了弱映射：

| 旧章节 | 新字段候选 |
|---|---|
| 事件概述 | `summary`、`timeline` |
| 关键事实 | `trigger_points`、`key_claims` |
| 传播机制 | `escalation_path`、`typical_escalation_path` |
| 情绪与立场 | `public_sentiment`、`related_emotions` |
| 风险点 | `indicators`、`trigger_conditions`、`early_warning_signals`、`failure_modes` |
| 可复用方法论 | `lessons_learned`、`recommended_tactics`、`actions` |
| 相似议题线索 | `example_cases` |

设计意义：

- 不强制重写旧 wiki。
- 尽量从旧文档中抽取可用字段。
- 保留字段覆盖率报告，用数据说明旧文档填充情况。
- 后续新领域资料按模板填写时，可以显著提升字段非空率。
