# 舆情事件分析改造计划（针对任务 `c93224fd-2842-425e-a90d-19cd023a84c7`）

## 目标与范围

本计划聚焦你本次反馈的 9 个问题，目标是把报告质量从“可用”提升到“稳定可复核”：

- 提升事件型检索的精准度，降低无关帖文混入。
- 修复 `topic_relevance_guard` 长期误判问题，建立可调可观测机制。
- 提升 Wiki 召回命中率与置信度控制，减少“召回到了但没用上”。
- 修复时间线跑题、渠道占比口径错误、地域分析文案误导。
- 改善理论研判“证据不足”反复出现的问题，降低模板化输出。
- 明确 Step9 与 Step10.4 的职责边界，减少重复输入成本。

## 问题分解（按你的反馈编号）

### 1) 关键词太长，导致采集误召回

**现象**
- 事件型 query 被扩展成较长词串，`data_num/data_collect` 命中范围过宽，混入无关帖子。

**当前代码线索**
- `workflow/event_analysis_pipeline.py` 的 `_normalize_search_words_for_collection`、`_derive_precise_event_search_words` 已有“短语提取”，但仍与其他扩展词混合并放大召回面。
- `workflow/netinsight_keywords.py` 在 normal 模式下会把多个词合并为 `;` 关系（OR），进一步放宽。

**改造动作**
- 事件型 query 增加“精确词优先模式”：
  - 当命中事件模板（如“X回应Y事件”）时，默认只保留 2~3 个高置信核心词（如：`铁路12306`、`家长孩子相隔14车厢`）。
  - 扩展词进入候补层，仅在数据量不足时再逐级放开（分层回退）。
- 新增采集策略开关（建议）：
  - `SONA_EVENT_QUERY_STRICT_MODE=true`
  - `SONA_EVENT_QUERY_FALLBACK_LEVELS=core,extended,broad`
- 在 `collect_plan` 展示“核心词 vs 扩展词”，让用户确认采用哪一层。

**验收标准**
- 同类事件任务中，前 100 条样本的主题相关率（人工抽样）> 75%。
- 核心事件关键词在 `命中关键词` 字段出现比例明显提升。

### 2) `topic_relevance_guard` 经常失败（`coverage=0.0 < 0.08`）

**现象**
- guard 与真实相关性不一致，导致误报失败。

**当前代码线索**
- `workflow/event_analysis_pipeline.py` 的 `_topic_relevance_metrics` 用 query anchor token 与 `keyword_stats` top keywords 求交集。
- 交集计算对词形变化、短语断裂、同义表达不鲁棒。

**改造动作**
- 把单一交集覆盖率升级为“组合评分”：
  - `coverage_token`：当前词交集（保留）。
  - `coverage_phrase`：核心短语命中（如“14车厢”“12306回应”）。
  - `coverage_field`：在原始 CSV 的 `标题/内容/命中关键词` 上做事件词命中率。
  - 综合分：`0.35*token + 0.35*phrase + 0.30*field`。
- 阈值自适应：
  - 事件型 query 用较低 token 阈值 + 较高 phrase 阈值。
  - 非事件型 query 维持当前策略。
- 输出更清晰诊断日志：
  - 哪一层失败、失败样本示例、推荐关键词修正建议。

**验收标准**
- 连续 20 次真实事件任务中，误报失败率 < 10%。
- 失败时日志能明确给出“改什么词可通过”。

### 3) Step10.2 `wiki_snapshot` 召回不准

**现象**
- Wiki 召回不稳定，未命中你需要的材料。

**当前代码线索**
- `workflow/wiki_cli.py` 目前有领域路由与 seed 注入，但默认 `topk=6`，且最终 `min_evidence_score` 过滤可能导致空集。

**改造动作**
- 事件类 query 的 wiki 召回增强：
  - `topk` 从 6 调整为动态 8~12（按 query 类型与证据密度）。
  - 引入“事件锚词强制召回”：包含 `12306/车厢/回应` 等锚词文档优先。
  - 召回后重排加入“事件短语完整命中”加权，避免泛理论文档抢位。
- 置信度策略升级：
  - 将“无结果”拆为“无召回”与“低置信被过滤”两类。
  - 输出 `dropped_candidates` 供调试。

**验收标准**
- 事件型 query 的 wiki 召回结果中，至少 2 条与事件主体直接相关（路径/标题可复核）。
- `_wiki_meta` 明确给出召回、过滤、最终保留数量。

### 4) 关键时间线节点抓到无关事件（如粤超）

**现象**
- `analysis_timeline` 主要按“时间词”过滤，未做事件相关性约束。

**当前代码线索**
- `tools/analysis_timeline.py` 的 `_filter_by_time_keywords` + `_extract_time_descriptions` 只看时间特征，不看事件锚词。

**改造动作**
- 时间线改成“双门槛”：
  - 门槛1：必须有时间线索。
  - 门槛2：必须满足事件相关词命中（至少命中核心词组之一）。
- 增加输入参数（由 pipeline 传入）：
  - `eventAnchorTerms`（如 `铁路12306`,`家长孩子相隔14车厢`）。
- 生成时间线前先做“相关性重排”，优先展示高相关事件节点。

**验收标准**
- 时间线 Top5 节点中，直接相关节点不少于 4 条。
- 无关节点占比显著下降（抽样评估）。

### 5) 渠道占比口径错误（变成 total count 问题）

**现象**
- 报告渠道占比未按各平台真实采集条数计算，展示偏差。

**当前代码线索**
- `workflow/event_analysis_pipeline.py` Step6.5 依赖 `platform_row_distribution`；若为空会 fallback 均分，导致失真。

**改造动作**
- 渠道占比强制改为“CSV 实际条数聚合”：
  - 直接从最终合并 CSV 的 `平台` 字段 group by 统计。
  - `platform_row_distribution` 仅作为兜底，不作为主数据源。
- 明确写入 `channel_distribution.json` 的 `calculation_source`：
  - `csv_groupby_platform` / `fallback_estimation`。

**验收标准**
- 报告渠道占比与 CSV 平台分布一致（允许 1 条以内误差）。
- 无 `fallback_estimation` 时不得出现均分数据。

### 6) 地域分析文案与实现口径不一致

**现象**
- 报告出现“未提取明确地域分布统计”，但实际已基于 `IP属地` 统计。

**当前代码线索**
- `tools/region_stats.py` 已按 `IP属地` 做统计；问题更可能在报告叙事层把有效结果误判成“证据不足”。

**改造动作**
- 在报告模板填充前，加入强制一致性映射：
  - 当 `region_stats.json.valid_rows_count > 0` 且 `top_provinces` 非空，禁止输出“未提取明确地域分布”类文案。
- 在 `report_meta` 增加口径字段：
  - `region_source = "IP属地"`。

**验收标准**
- 有 `top_provinces` 数据时，地域小节必须输出具体省份与占比。
- 文案不得与统计结果冲突。

### 7) 理论研判仍频繁“证据不足”，且三理论模板化重复

**现象**
- 理论槽位重复“证据不足/沉默螺旋/议程设置/蝴蝶效应”。

**当前代码线索**
- `tools/report_html_template.py` 有 fallback 理论逻辑，仍可能在证据映射不足时触发模板化填充。
- `prompt/report_html_template_fill.txt` 虽要求多理论来源，但落地约束还不够硬。

**改造动作**
- 理论生成改为“先选证据、后写理论”：
  - 从 `reference_insights` / `oprag_snapshot` / `wiki_snapshot` 先挑选候选理论条目；
  - 至少有 2 条理论证据时，禁止输出“证据不足”。
- 增加“去重复理论槽位”校验器：
  - 三个槽位理论名不能重复；重复则自动二次生成该段。
- 降低对单一方法论文档的依赖：
  - 方法论文档作为“框架提示”，理论事实以事件召回证据优先。

**验收标准**
- 理论槽位中“证据不足”出现率显著下降（事件有召回时应趋近 0）。
- 三理论槽位名称不重复，且每条有来源映射。

### 8) “舆论引导与处置建议 / 总结复盘”质量较好

**动作**

- 在后续生成中继续复用其结构风格（但保持数据证据约束）。

### 9) Step9 与 Step10.4 都让输入研判重点，职责重叠

**现状区别**
- Step9（`weibo_aisearch + user_judgement`）：
  - 输入是“针对当前事件本次报告的即时研判重点”，直接参与本次报告主线推理。
- Step10.4（`reference_insights` 的专家补充）：
  - 输入更偏“补充参考材料/专家视角”，应作为辅助证据，不应覆盖主线判断。

**改造动作**
- UI 与提示文案拆分明确：
  - Step9 改名：`本次事件主研判（主输入）`
  - Step10.4 改名：`可选专家补充（辅助输入）`
- 增加“复用上一输入”选项：
  - Step10.4 可一键引用 Step9 内容再编辑，避免重复录入。
- 在 `user_judgement_input.json` 和 `user_expert_notes.json` 增加 `source_from` 字段标识关联。

**验收标准**
- 用户只输入一次也可完整走通（第二步可跳过或一键复用）。
- 报告中主线与补充来源角色清晰。

## 风险与注意事项

- 关键词收紧后可能出现“采集总量不足”，需保留分层回退机制。
- topic guard 阈值放宽不能导致真正跑题漏检，必须通过组合评分平衡。
- wiki 召回增强会增加 token 消耗，需设置 topk 与上下文预算上限。

## 结论

你这次 60 分判断是准确的：核心短板在“检索与结构化中间层”，不是最终文案层。  
本计划优先修中间层的“词、召回、守卫、口径一致性”，再做报告生成层的理论质量与交互体验优化。这样能最快把报告稳定拉到可用区间（预计先到 75~80 分），再逐步提升到高质量区间。
